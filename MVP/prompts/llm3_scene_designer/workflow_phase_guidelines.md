# workflow_step 场景设计约束（LLM3 必须遵守）

输入中的当前 scene 来自 LLM2 scene planner，并且会包含：

- `workflow_step`
- `scene_goal`
- `entry_requirement`
- `key_points`
- `scene_outputs`
- `handoff_to_next`
- `layout_prompt`
- `panels`
- `beat_sequence`

其中，`workflow_step` 不是参考信息，而是当前 scene 的教学职责约束。

你必须把它当作“这一幕为什么存在、应该怎么讲、哪些内容不该放进来”的硬信号。

## 总原则

1. 先服从 `workflow_step`，再做视觉设计。
2. 不要把不属于当前 step 的职责混进这一幕。
3. `scene_goal` 决定本幕要完成什么，`key_points` 决定本幕必须讲清哪些核心点。
4. `entry_requirement` 决定本幕开场的认知前提，`scene_outputs` 与 `handoff_to_next` 决定本幕结束时必须交出去什么。
5. `layout_prompt` 决定本幕的大体信息组织方式。
6. `panels` 与 `beat_sequence` 是本幕的面板级与节拍级合同；你必须尊重它们。
7. `workflow_step` 决定的是教学角色，不是 object 生命周期本身；但在本项目里所有 scene 都必须对象独立，`entry_state / exit_state` 都应为空。

## step -> scene 设计职责

### `problem_intake`

职责：

- 完整、清楚地呈现题目、问法、题图、单位与任务边界
- 让观众知道这是几问、最终要算哪些量

设计要求：

- 以读题和题意理解为主
- 可以重绘题图、净化题图、列出已知/未知
- 可以建立后续会长期复用的主图骨架

明确禁止：

- 不要展开正式推导
- 不要把本幕做成复杂动画主导的过程展示

### `preview`

职责：

- 先让观众形成全局直觉：起点、过程、关键事件、终点

设计要求：

- 以完整过程的快速预演为主
- 允许更强的运动展示与轨迹回放
- 叙事上强调“先看懂发生了什么”，而不是“现在就推导为什么”

明确禁止：

- 不要承担正式公式推导
- 不要把多步计算链塞进 preview

### `goal_lock`

职责：

- 锁定“这一问到底要求什么”
- 从题干中筛出与当前问题直接相关的条件

设计要求：

- 结论应聚焦、简短、明确
- 可以用局部高亮、题干缩略、条件摘录帮助观众进入当前问题

明确禁止：

- 不要在本幕完成完整建模 + 推导 + 计算全套流程

### `model`

职责：

- 把语言条件翻译成图、符号、对象、约束、坐标系、参考方向

设计要求：

- 主图必须稳定、清楚、可供后续推导复用
- 允许加入坐标系、示意箭头、边界条件、变量命名
- 本幕的核心是“把条件变成模型”

明确禁止：

- 不要退化成只看运动过程的大图展示
- 不要把完整推导链提前塞进建模幕

### `method_choice`

职责：

- 明确采用哪条主线求解，以及为什么这样选

设计要求：

- 可以出现少量关键公式或方法比较
- 重点是“选法”和“理由”，不是长计算

明确禁止：

- 不要把 method_choice 变成完整 derive
- 不要让画面失去与主图的关联

### `derive`

职责：

- 完成核心推导与计算，得到关键中间量或最终结果

设计要求：

- 必须有稳定主图作为参照
- 公式链应分步呈现，每一步只承担一个新结论
- 关键中间量要命名并体现物理/数学意义

明确禁止：

- 不要把本幕做成只有大图动画、几乎没有可跟随的推导
- 不要把推导写成一屏倾倒式长公式

### `check`

职责：

- 对结果做量纲、极限、符号、方向、物理意义一致性检查

设计要求：

- 以验证和解释为主
- 可以引用上一幕结果，但不应重新展开整套求解

明确禁止：

- 不要在 check 幕里重新做大量新推导

### `recap`

职责：

- 回顾本问结论，或把多问结果收口到统一结论板

设计要求：

- 可以用简洁的结果面板、重点回顾、短动画回放
- 重点是收束信息、准备承接下一幕或结束视频

明确禁止：

- 不要在 recap 幕引入大量新变量和新推导

### `transfer`

职责：

- 提炼方法模板、易错点、变式迁移

设计要求：

- 从“一次性解题”提升到“可复用技能”
- 适合用总结面板、方法 bullet、反例提醒、变式提示

明确禁止：

- 不要把 transfer 幕重新讲成一道新题的求解过程

## 设计时的优先判断

如果 `workflow_step` 是：

- `preview`：优先“看懂过程”，不要正式推导
- `problem_intake` / `goal_lock`：优先“看清题、锁定任务”，不要过度展开
- `model` / `method_choice`：优先“建立解题框架”，不要抢跑到大段计算
- `derive`：优先“稳定主图 + 分步推导”
- `check`：优先“验证与解释”
- `recap` / `transfer`：优先“收束、总结、迁移”

## 与 layout / motion 的关系

1. `workflow_step` 会影响你如何分配画面重点，但不改变 `layout_contract` 与 `motion_contract` 必须输出的要求。
2. 如果是 `preview`，可以让 motion 更突出；如果是 `derive`，应让主图与推导并存且主次清楚。
3. 如果 `workflow_step` 与 `scene_goal / key_points` 冲突，以“step 决定角色、goal 决定本幕具体内容”的方式协调，不要无视 step。
4. 如果 `layout_prompt` 与你自己的直觉冲突，优先服从 `layout_prompt`。
5. 如果 `beat_sequence` 已经规定了“先看题、再预演、再冻结、再过渡”，你不能把它改写成另一种讲法。
